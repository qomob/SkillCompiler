# Parallel Extractors — 并行知识提取器

**加载时机：** 📍 Pass 2 Extract 执行知识提取时加载（当 source 来自多源摄取或内容复杂时）。

---

## 设计哲学

从长内容（整份 PDF、视频转写、网页文档）提取知识时，单线程逐句扫描效率低且容易遗漏。并行提取器的核心：**派多个专项提取器同时扫描同一份内容，每个只关注一类知识，最后合并去重**。

类比编译器的并行优化 pass：每个 pass 只做一个具体的优化（常量折叠、死代码消除），但它们可以并行跑，最后合并 IR。

---

## 五个专项提取器

| 提取器 | 关注的知识类型 | 输出 |
|--------|--------------|------|
| **F1 框架提取器** | 方法论、流程、步骤模型、决策树 | 结构化的流程描述 |
| **F2 原则提取器** | 规则、约束、最佳实践、戒律 | 规则清单 |
| **F3 案例提取器** | 具体实例、应用场景、成功案例 | 带上下文的案例 |
| **F4 反例提取器** | 失败案例、常见错误、反模式、边界外的行为 | 负面清单 |
| **F5 术语提取器** | 领域概念、专有名词、定义、术语表 | 术语词典 |

### 为什么是这五类

这五类覆盖了 skill 所需知识的完整光谱：

- **F1 框架** 回答"怎么做"（Workflow）
- **F2 原则** 回答"必须遵守什么"（Rules）
- **F3 案例** 回答"什么时候用"（Examples）
- **F4 反例** 回答"什么时候不该用"（Boundary）
- **F5 术语** 回答"关键词什么意思"（Domain Knowledge）

对应 skill 架构：F1→workflows，F2→references/rules，F3→examples，F4→honest-boundaries/gotchas，F5→references/domain。

---

## 执行机制

### 并行触发条件

当满足任一条件时，启用并行提取器（而非 Pass 2 的默认单线程提取）：

| 条件 | 理由 |
|------|------|
| Pass I 执行了（多源摄取） | 摄取后的内容通常量大且杂 |
| `structured_content` > 3000 字 | 单线程扫描长内容易遗漏 |
| 内容含 3+ 个独立主题 | 需要多维度并行覆盖 |
| 用户显式要求深度提取 | 质量优先 |

### 并行执行模型

五个提取器**逻辑并行**（在同一轮中分别扫描全文），每个独立产出候选知识列表：

```
structured_content
      │
  ┌───┼───┬───┬───┬───┐
  F1  F2  F3  F4  F5
  │   │   │   │   │
  ▼   ▼   ▼   ▼   ▼
 候选  候选  候选  候选  候选
  │   │   │   │   │
  └───┴───┴───┴───┴───┘
      │
  合并去重 + 三重验证
      │
      ▼
  validated knowledge_inventory
```

## 各提取器规格

### F1 — 框架提取器

**扫描目标：** 方法论、流程、步骤性内容。

**识别信号：**
- 编号步骤（"第一步...第二步..."）
- 流程图/决策树的文字描述
- "流程是"、"按以下步骤"、"方法论包括"
- 时序性叙述（先...再...最后...）

**产出格式：**
```json
{
  "extractor": "F1-framework",
  "candidates": [
    {
      "content": "代码审查三阶段：自动检查 → 人工审查 → 安全审计",
      "source": "SEG-014",
      "evidence": "primary",
      "framework_steps": ["自动检查", "人工审查", "安全审计"]
    }
  ]
}
```

### F2 — 原则提取器

**扫描目标：** 规则、约束、强制要求。

**识别信号：**
- 强制词："必须"、"禁止"、"不要"、"always"、"never"、"强制"
- 建议词："应该"、"建议"、"推荐"、"避免"
- 约束词："不得超过"、"至少需要"、"限定为"

**产出格式：**
```json
{
  "extractor": "F2-principle",
  "candidates": [
    {
      "content": "禁止硬编码 API key 在源码中",
      "source": "SEG-022",
      "evidence": "primary",
      "rule_type": "prohibition"
    }
  ]
}
```

### F3 — 案例提取器

**扫描目标：** 具体实例、应用场景。

**识别信号：**
- "例如"、"示例"、"case"、"场景"
- 具体的输入输出对
- "当...时，应该..."

**产出格式：**
```json
{
  "extractor": "F3-case",
  "candidates": [
    {
      "content": "处理用户登录时，先验证 CSRF token 再查数据库",
      "source": "SEG-031",
      "evidence": "primary",
      "case_type": "application"
    }
  ]
}
```

### F4 — 反例提取器

**扫描目标：** 失败案例、反模式、常见错误。

**识别信号：**
- "不要"、"错误"、"避免"、"反模式"、"常见错误"
- "踩坑"、"教训"、"失败的"
- 负面表述（"这样会导致..."）

**产出格式：**
```json
{
  "extractor": "F4-antipattern",
  "candidates": [
    {
      "content": "不要在循环中创建数据库连接，会导致连接池耗尽",
      "source": "SEG-040",
      "evidence": "primary",
      "antipattern_type": "performance"
    }
  ]
}
```

**重要：** F4 的产出直接喂给 [honest-boundaries.md](honest-boundaries.md) 的失败模式章节。

### F5 — 术语提取器

**扫描目标：** 领域概念、专有名词。

**识别信号：**
- 定义句（"X 是指..."、"X 定义为..."）
- 首字母缩写（CSRF、SSRF、XSS）
- 反复出现的领域名词
- 中英文对照术语

**产出格式：**
```json
{
  "extractor": "F5-term",
  "candidates": [
    {
      "content": "CSRF（Cross-Site Request Forgery）：跨站请求伪造",
      "source": "SEG-005",
      "evidence": "primary",
      "term": "CSRF"
    }
  ]
}
```

## 合并去重

五个提取器产出后，合并为统一候选池：

### 去重规则

| 情况 | 处理 |
|------|------|
| 同一内容被多个提取器命中 | 保留证据等级最高的版本，合并 source 标注 |
| 语义相同表述不同 | 保留表述最清晰的版本，记录变体 |
| 矛盾内容（F2 说"必须" vs 另一来源说"可选"） | 不去重，触发 [evidence-grading.md](evidence-grading.md) 冲突保留规则 |

### 去重后

候选池进入**三重验证**。

---

## 三重验证

**核心原则：** 不是所有候选都值得变成 skill 知识。每个候选必须通过三项检验才能进入 `knowledge_inventory`。

### V1 — 独立佐证（Corroboration）

候选知识必须在原始内容中有 **≥ 2 处独立佐证**，或来自 `primary` 证据且表述明确。

| 通过条件 | 说明 |
|---------|------|
| 同一观点出现 2+ 次（不同段落） | 强佐证 |
| 1 次 primary + 1 次 secondary | 中佐证，通过 |
| 仅 1 次 inferred | **不通过**（孤立推断不可靠） |

### V2 — 预测力（Predictive Power）

候选知识必须能回答原始内容中**未明说的新问题**。

**测试方法：** 构造一个原始内容未直接讨论的场景，看这条知识是否能给出指导。若不能（即只是重复原文事实），则预测力不足。

| 通过条件 | 说明 |
|---------|------|
| 能指导原文未覆盖的新场景 | 通过 |
| 仅在原文特定上下文成立 | 降级为 example（而非 rule） |
| 纯事实陈述无泛化能力 | **不通过**（不是可复用知识） |

### V3 — 非常识性（Non-triviality）

候选知识不应是"所有从业者都知道"的常识。

| 通过条件 | 说明 |
|---------|------|
| 包含领域特定的非显而易见洞察 | 通过 |
| 是通用编程/写作常识（如"变量要命名清晰"） | **不通过**（无增值） |
| 是原始内容的独特观点 | 通过 |

### 验证结果

| 通过几项 | 处理 |
|---------|------|
| V1 + V2 + V3 全通过 | 进入 `knowledge_inventory` |
| 通过 2 项 | 降级为 `examples`（参考用，非规则） |
| 通过 ≤ 1 项 | 淘汰 |

**典型通过率：** 经验上，候选池中只有 25-50% 能通过三重验证。这是正常的——高门槛保证 skill 质量不注水。

## Output

并行提取器的产出合并进 Pass 2 的标准 `knowledge_inventory` 结构，每条知识额外携带：

```json
{
  "extractor": "F1-framework",
  "verification": {
    "v1_corroboration": true,
    "v2_predictive": true,
    "v3_non_trivial": true,
    "passed": true
  }
}
```

## Gotchas

1. **并行提取器不是必须的** — 纯文本短 Prompt 直接用 Pass 2 默认单线程提取即可。强制对 200 字 Prompt 跑 5 个提取器是过度设计。
2. **F4 反例提取器最易被忽略但最有价值** — 反例直接喂给诚实边界，是 skill 可信度的关键。不要因为"负面内容少"就跳过 F4。
3. **三重验证的 V2 最难判定** — "预测力"需要构造假设场景测试，容易被跳过。但跳过 V2 会导致 skill 充满不可复用的碎片事实。
4. **去重不要在提取器内部做** — 每个提取器只管自己那类，跨类去重在合并阶段统一做。提取器内部去重会丢失跨类关联。
5. **通过率低是好事** — 若通过率 > 70%，说明验证门槛太松，skill 会注水。调整验证标准而非降低门槛。
